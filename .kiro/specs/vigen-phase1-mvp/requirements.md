# Requirements Document

## Introduction

VIGEN is a music-driven AI video generation SaaS. Users upload an audio track (MP3/WAV/FLAC) and receive a cinematic, beat-synchronized music video. The core pipeline is: Audio Upload → Analysis → Storyboard Generation → Storyboard Approval Gate → Parallel Scene Rendering → FFmpeg Assembly → Download/Share.

The current codebase has a well-structured scaffold (Sprint 1 level) with correct state machines, API shapes, database models, and WebSocket plumbing. However, every AI/ML integration and every media processing step is either a mock stub or incomplete. This requirements document covers the full Phase 1 MVP — what must be built, completed, or wired to deliver a production-ready product.

**Current implementation status (as of March 2026):**
- ✅ Auth endpoints, JWT, DB models, Alembic migration, Docker Compose
- ✅ Job state machine (10 states), WebSocket progress, scene retry endpoints
- ✅ Frontend shell: landing, login/register, dashboard, job detail, storyboard review
- ✅ Mock pipeline mode for frontend-only development
- ⚠️ Audio analysis uses librosa + Gemini stub (Gemini call is text-only, not multimodal)
- ⚠️ Storyboard generation calls Claude via LiteLLM but has no Haiku enrichment pass
- ⚠️ Video rendering calls Kling via LiteLLM but the response parsing is unverified
- ⚠️ FFmpeg assembly is implemented but uses blocking subprocess and no duration validation
- ⚠️ Storage supports R2 but falls back to filesystem; partial preview writes placeholder bytes
- ⚠️ Celery tasks exist but analysis/planning run synchronously in the API request thread
- ❌ Hard cost cap enforcement missing
- ❌ CI/CD pipeline missing
- ❌ Credit service not implemented
- ❌ SendGrid integration is a log-only stub
- ❌ Create job page (`/create`) is empty
- ❌ Style preset selection not wired in frontend
- ❌ `create_all` in lifespan bypasses Alembic

## Glossary

- **VIGEN**: The AI music video generation SaaS platform being built
- **Job**: A single end-to-end video generation request, tracked through a 10-state state machine
- **Scene**: A single video clip segment within a job, with its own timing, prompt, and render status
- **Storyboard**: A structured JSON document containing all scenes for a job, generated by Claude
- **Beat Map**: Extracted audio metadata (BPM, beat times, onset times, bar times) stored per job
- **Provider**: An abstracted interface for AI/ML services (audio analysis, storyboard generation, video rendering)
- **Celery**: The distributed task queue used for async background processing
- **LiteLLM**: The unified LLM routing library used to call Claude, Gemini, and other models
- **Kling**: The primary video generation API (Kling 1.6) used for scene rendering
- **FFmpeg**: The media processing tool used for video assembly and audio mapping
- **R2**: Cloudflare R2 object storage used for audio uploads, rendered clips, and final videos
- **CDN**: Cloudflare CDN used to serve media assets
- **Credit**: A unit of platform currency consumed per job render
- **Approval Gate**: The mandatory user review step before any GPU/rendering spend is incurred
- **Partial Preview**: A low-resolution assembled video made available after the first N scenes complete
- **SendGrid**: The email delivery service used for job completion notifications
- **structlog**: The structured logging library used for correlation-ID-tagged log output
- **Alembic**: The database migration tool; the only permitted mechanism for schema changes
- **MSW**: Mock Service Worker — browser-level API mocking for frontend development
- **TanStack Query**: React server-state management library used in the frontend
- **Zustand**: Lightweight React client-state management library

## Requirements

### Requirement 1 — Celery Async Architecture

**User Story:** As a backend engineer, I want all long-running pipeline stages to run in Celery workers rather than in the API request thread, so that the API server remains responsive and render jobs do not time out.

#### Acceptance Criteria

1. WHEN a job is created via `POST /api/jobs`, THE System SHALL enqueue a Celery task (`jobs.analyse_and_plan`) and return the job record with `status="queued"` without waiting for analysis to complete.
2. WHEN the `jobs.analyse_and_plan` Celery task executes, THE System SHALL run audio analysis and storyboard generation entirely within the worker process.
3. WHEN the `jobs.render` Celery task executes, THE System SHALL run all scene rendering and final assembly entirely within the worker process.
4. WHEN a Celery task fails with an unhandled exception, THE System SHALL set the job `status` to `"failed"` and record the exception message in `failure_reason`.
5. WHILE a Celery task is executing, THE System SHALL publish `ProgressEvent` messages to the WebSocket broker so the frontend receives live updates.

---

### Requirement 2 — Real Audio Analysis Pipeline

**User Story:** As a creator, I want VIGEN to accurately extract BPM, beat times, and mood from my uploaded audio, so that the generated storyboard is genuinely synchronized to my music.

#### Acceptance Criteria

1. WHEN the analysis worker receives a job, THE System SHALL load the audio file using `librosa` at 22050 Hz sample rate and extract BPM, beat frame times, and onset times.
2. WHEN librosa analysis completes, THE System SHALL call Gemini 2.0 Flash with the audio file content (multimodal) to extract mood, genre, key, and a one-sentence summary.
3. IF the Gemini API call fails, THEN THE System SHALL log the error and use a deterministic fallback derived from librosa energy features rather than hardcoded values.
4. WHEN analysis completes, THE System SHALL persist the beat map to the `audio_beat_maps` table with accurate `bpm`, `beat_times`, `bar_times`, `downbeat_times`, and `onset_times` fields.
5. WHEN analysis completes, THE System SHALL store the full analysis result in `job.audio_analysis` as a JSON object containing `bpm`, `beats`, `onsets`, `mood`, `genre`, `key`, `energy_arc`, and `scene_count_hint`.

---

### Requirement 3 — Real Storyboard Generation

**User Story:** As a creator, I want VIGEN to generate a detailed, beat-aligned storyboard using Claude, so that each scene has a meaningful visual prompt and the total duration matches my audio.

#### Acceptance Criteria

1. WHEN the planning stage begins, THE System SHALL call Claude Sonnet 4.5 via LiteLLM with a prompt that includes BPM, mood, genre, energy arc, and target scene count derived from the audio analysis.
2. WHEN Claude returns a storyboard JSON, THE System SHALL validate it against the `StoryboardSchema` Pydantic model, including contiguous scene times and total duration matching `audio_duration_seconds` within 0.1 seconds.
3. IF Claude returns invalid JSON or a schema-invalid storyboard, THEN THE System SHALL retry the LiteLLM call up to 2 times before falling back to the deterministic scene generator.
4. WHEN the storyboard is valid, THE System SHALL persist all scenes to the `scenes` table and set `job.status` to `"awaiting_approval"`.
5. WHEN a storyboard is rejected by the user with a reason, THE System SHALL include the rejection reason in the next Claude prompt as a revision instruction.
6. THE System SHALL capture the `prompt_template_version` on each job at the time of storyboard generation.

---

### Requirement 4 — Storyboard Approval Gate

**User Story:** As a creator, I want to review and approve the storyboard before any rendering begins, so that I control GPU spend and can reject weak outputs.

#### Acceptance Criteria

1. WHEN a job reaches `status="awaiting_approval"`, THE System SHALL NOT dispatch any render tasks until the user explicitly calls `POST /api/jobs/{id}/approve`.
2. WHEN a user approves a storyboard, THE System SHALL set `storyboard_approved=true`, record `storyboard_approved_at`, transition `status` to `"rendering"`, and enqueue the `jobs.render` Celery task.
3. WHEN a user rejects a storyboard with a reason, THE System SHALL increment `storyboard_regeneration_count`, store the rejection reason, and re-enqueue `jobs.analyse_and_plan`.
4. WHEN the storyboard review page loads, THE System SHALL display each scene's `visual_prompt`, `mood`, `camera_angle`, `motion_type`, `color_palette`, and timing range.
5. WHEN the storyboard review page loads, THE System SHALL display the pre-render cost estimate prominently before the approve button.

---

### Requirement 5 — Parallel Scene Rendering

**User Story:** As a creator, I want scenes to render in parallel rather than sequentially, so that my video completes faster.

#### Acceptance Criteria

1. WHEN the render task begins, THE System SHALL dispatch each pending scene as an independent Celery sub-task using a `group` or `chord` pattern rather than a sequential loop.
2. WHEN a scene render sub-task executes, THE System SHALL call the Kling 1.6 API via the `VideoRenderProvider` with the scene's `visual_prompt` and duration.
3. WHEN a scene render completes, THE System SHALL store the returned video clip URL in `scene.video_url`, record `cost_usd` and `render_time_seconds`, and set `scene.status="complete"`.
4. IF a scene render fails, THEN THE System SHALL set `scene.status="failed"`, record the error, and continue rendering remaining scenes rather than aborting the job.
5. WHEN the number of completed scenes reaches `preview_scene_threshold` (default 5), THE System SHALL trigger a partial preview assembly using FFmpeg and store the result in `job.preview_video_url`.
6. WHILE rendering is in progress, THE System SHALL enforce the per-job `cost_cap_usd` after each scene completes and set remaining scenes to `"failed"` with reason `"cost_cap_exceeded"` if the cap is reached.

---

### Requirement 6 — FFmpeg Assembly

**User Story:** As a creator, I want the final video to be a properly assembled MP4 with my original audio track, so that the output is a watchable, synchronized music video.

#### Acceptance Criteria

1. WHEN all scenes for a job are complete, THE System SHALL run FFmpeg with `-f concat -safe 0` to concatenate all scene clips in scene index order.
2. WHEN running FFmpeg assembly, THE System SHALL map the original uploaded audio track using `-map 1:a -c:a aac -shortest` to replace any audio from the rendered clips.
3. WHEN FFmpeg assembly completes, THE System SHALL validate that the final video duration is within 0.5 seconds of `job.audio_duration_seconds`.
4. IF FFmpeg assembly fails, THEN THE System SHALL set `job.status="failed"` with a descriptive `failure_reason` and log the full FFmpeg stderr output.
5. WHEN the final video is assembled, THE System SHALL upload it to the configured storage backend (R2 or filesystem) and store the public URL in `job.final_video_url`.
6. WHEN partial preview assembly is triggered, THE System SHALL run the same FFmpeg concat process on the first N completed scenes and store the result in `job.preview_video_url`.

---

### Requirement 7 — Object Storage Integration

**User Story:** As a platform operator, I want all media assets stored in Cloudflare R2 rather than the local filesystem, so that the platform is stateless and horizontally scalable.

#### Acceptance Criteria

1. WHEN `storage_backend="r2"` is configured, THE System SHALL upload audio files, rendered scene clips, preview videos, and final videos to the R2 bucket using the `aioboto3` S3-compatible client.
2. WHEN a file is uploaded to R2, THE System SHALL return a CDN-served public URL using the `r2_public_base_url` configuration value.
3. WHEN `storage_backend="filesystem"` is configured, THE System SHALL write files to the local `filesystem_storage_root` path and serve them via the `/media` static mount.
4. WHEN the audio upload endpoint receives a file, THE System SHALL validate that the file is MP3, WAV, or FLAC and does not exceed 200 MB before writing to storage.
5. IF an R2 write fails, THEN THE System SHALL log the error with the file path and raise an exception that causes the Celery task to retry.

---

### Requirement 8 — Real-Time Progress via WebSocket

**User Story:** As a creator, I want to see live progress updates while my video is rendering, so that I know the job is progressing and can estimate when it will complete.

#### Acceptance Criteria

1. WHEN a client connects to `GET /api/jobs/{id}/ws`, THE System SHALL accept the WebSocket connection and subscribe it to progress events for that job.
2. WHEN a pipeline stage transitions (analysis, planning, rendering, assembly, done), THE System SHALL publish a `ProgressEvent` with `stage`, `percent`, `scenes_complete`, `scenes_total`, and `message` fields.
3. WHEN a scene render completes, THE System SHALL publish a `ProgressEvent` with updated `scenes_complete` count.
4. WHEN the job reaches `status="complete"`, THE System SHALL publish a final `ProgressEvent` with `stage="done"` and `percent=100`.
5. WHEN the WebSocket connection is closed by the client, THE System SHALL clean up the subscription without affecting the running job.

---

### Requirement 9 — Job Completion Email

**User Story:** As a creator, I want to receive an email when my video is ready, so that I can step away during long renders and return when it's done.

#### Acceptance Criteria

1. WHEN a job reaches `status="complete"`, THE System SHALL call `NotificationService.send_job_complete_email` with the user's email, job ID, and final video URL.
2. WHEN `sendgrid_api_key` is configured, THE System SHALL send the completion email via the SendGrid API with the job ID and a direct download link.
3. IF the SendGrid API call fails, THEN THE System SHALL log the error and continue without retrying, so that email failure never causes job failure.
4. WHEN `sendgrid_api_key` is not configured, THE System SHALL log the notification details at INFO level as a no-op fallback.

---

### Requirement 10 — Credit Ledger Foundation

**User Story:** As a platform operator, I want a credit ledger that tracks consumption per job, so that I can validate unit economics and prepare for billing in Phase 2.

#### Acceptance Criteria

1. WHEN a job is approved for rendering, THE System SHALL call `CreditService.deduct_credits` to reserve the estimated cost from the user's credit balance.
2. WHEN a job completes, THE System SHALL call `CreditService.reconcile_credits` to adjust the reservation to the actual cost, crediting any unused amount back.
3. WHEN a credit deduction is made, THE System SHALL insert a record into the `credit_transactions` table with `user_id`, `job_id`, `amount`, `type`, and `created_at`.
4. IF a user has insufficient credits to cover the estimated cost, THEN THE System SHALL reject the approval request with a `402 Payment Required` response.
5. THE System SHALL expose `GET /api/credits/balance` to return the authenticated user's current credit balance.

---

### Requirement 11 — Per-Job Hard Cost Cap

**User Story:** As a platform operator, I want each job to have an enforced cost cap, so that a runaway render cannot exceed the user's budget or the platform's cost tolerance.

#### Acceptance Criteria

1. WHEN a job is created, THE System SHALL set `cost_cap_usd` to the user-provided value or the system default of `$20.00`.
2. WHEN a scene render completes and `job.actual_cost_usd` exceeds `job.cost_cap_usd`, THE System SHALL immediately set all remaining `pending` scenes to `status="failed"` with `failure_reason="cost_cap_exceeded"`.
3. WHEN the cost cap is triggered, THE System SHALL set `job.status="failed"` with `failure_reason="cost_cap_exceeded"` and proceed to assemble whatever scenes completed successfully.
4. WHEN the cost cap is triggered, THE System SHALL publish a `ProgressEvent` notifying the client that the cap was reached.

---

### Requirement 12 — Style Preset Selection

**User Story:** As a creator, I want to choose a visual style preset when creating a job, so that the generated scenes reflect my aesthetic preferences.

#### Acceptance Criteria

1. WHEN the create job page loads, THE System SHALL fetch and display all available style presets from `GET /api/presets`.
2. WHEN a user selects a preset, THE System SHALL include the `style_preset_id` in the `POST /api/jobs` request body.
3. WHEN the storyboard is generated, THE System SHALL inject the selected preset's `prompt_modifier`, `motion_bias`, and `lighting_bias` into the Claude prompt.
4. WHEN no preset is selected, THE System SHALL use a default neutral preset for storyboard generation.

---

### Requirement 13 — Create Job Page

**User Story:** As a creator, I want a clear, guided job creation flow, so that I can upload my audio and start a render without confusion.

#### Acceptance Criteria

1. WHEN a user navigates to `/create`, THE System SHALL display a drag-and-drop audio upload zone that accepts MP3, WAV, and FLAC files up to 200 MB.
2. WHEN a user drops or selects a valid audio file, THE System SHALL upload it via `POST /api/uploads/audio` and display upload progress.
3. WHEN the upload completes, THE System SHALL display the style preset selector and a "Create job" button.
4. WHEN the user clicks "Create job", THE System SHALL call `POST /api/jobs` with the audio URL, file name, detected duration, and selected preset ID, then redirect to `/job/{id}`.
5. IF the upload fails or the file is invalid, THEN THE System SHALL display a clear error message and allow the user to retry without losing their preset selection.

---

### Requirement 14 — Dashboard and Job History

**User Story:** As a creator, I want to see all my past and in-progress jobs in a dashboard, so that I can track status, download completed videos, and navigate to jobs that need my attention.

#### Acceptance Criteria

1. WHEN an authenticated user navigates to `/dashboard`, THE System SHALL fetch and display all jobs for that user ordered by `created_at` descending.
2. WHEN a job has `status="awaiting_approval"`, THE System SHALL display a prominent "Review storyboard" action link on the job card.
3. WHEN a job has `status="complete"`, THE System SHALL display a download button linked to `final_video_url`.
4. WHEN a job has `status="failed"`, THE System SHALL display the `failure_reason` on the job card.
5. WHEN the dashboard loads with no jobs, THE System SHALL display an empty state with a call-to-action to create the first job.

---

### Requirement 15 — Scene Retry

**User Story:** As a creator, I want to retry individual failed scenes without re-rendering the entire job, so that I can recover from partial failures efficiently.

#### Acceptance Criteria

1. WHEN a user calls `POST /api/jobs/{id}/scenes/{n}/retry`, THE System SHALL reset the scene `status` to `"pending"`, increment `regeneration_count`, and enqueue a targeted render task for that scene only.
2. WHEN a scene retry is triggered, THE System SHALL set `job.status` back to `"rendering"` if the job was previously `"complete"` or `"failed"`.
3. WHEN the retried scene completes, THE System SHALL re-run final assembly if all other scenes are also complete.
4. WHEN the job detail page shows a failed scene, THE System SHALL display a "Retry scene" button on that scene card.

---

### Requirement 16 — Structured Logging and Observability

**User Story:** As a platform operator, I want structured logs with correlation IDs on every pipeline event, so that I can diagnose failures and track cost/latency per job.

#### Acceptance Criteria

1. WHEN any pipeline stage executes, THE System SHALL emit a `structlog` log entry containing `job_id`, `scene_id` (where applicable), `stage`, `provider`, `model`, `latency_ms`, and `cost_usd`.
2. WHEN a Celery task starts or completes, THE System SHALL log the task name, job ID, and wall-clock duration.
3. WHEN an external API call (Gemini, Claude, Kling, SendGrid) is made, THE System SHALL log the provider name, model, request latency, and whether the call succeeded or fell back.
4. WHEN a job transitions between states, THE System SHALL log the old state, new state, job ID, and timestamp.

---

### Requirement 17 — Database Migration Discipline

**User Story:** As a backend engineer, I want all schema changes to go through Alembic migrations, so that production databases can be upgraded safely without data loss.

#### Acceptance Criteria

1. THE System SHALL NOT call `Base.metadata.create_all` in the application lifespan or anywhere outside of test fixtures.
2. WHEN the application starts, THE System SHALL verify that the database schema is at the latest Alembic revision and log a warning if it is not.
3. WHEN a new model field is added, THE System SHALL include a corresponding Alembic migration file that can be applied and rolled back cleanly.

---

### Requirement 18 — CI/CD Pipeline

**User Story:** As a developer, I want an automated CI pipeline that runs on every push, so that regressions are caught before they reach staging.

#### Acceptance Criteria

1. WHEN code is pushed to any branch, THE System SHALL run Python linting (`ruff`), type checking, and `python -m compileall` on the backend.
2. WHEN code is pushed to any branch, THE System SHALL run `node --check` and Next.js build validation on the frontend.
3. WHEN code is pushed to the main branch, THE System SHALL build Docker images for the API, worker, and frontend services.
4. WHEN all CI checks pass on the main branch, THE System SHALL produce build artifacts ready for staging deployment.

---

### Requirement 19 — Frontend Bug Fixes and Import Corrections

**User Story:** As a frontend developer, I want all import paths and component references to be correct, so that the application builds and runs without errors.

#### Acceptance Criteria

1. WHEN the frontend application builds, THE System SHALL resolve all component imports without errors (e.g., `ui.jsxx` references must be corrected to `ui.jsx`).
2. WHEN the `ClientShell` component renders, THE System SHALL import `Topbar` from the correct `ui.jsx` path.
3. WHEN the create job page (`/create`) is loaded, THE System SHALL render a functional upload and job creation form rather than an empty page.
4. WHEN any page imports `React`, THE System SHALL only import it where JSX is used, removing unused imports that cause lint warnings.
